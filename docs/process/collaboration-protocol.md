# Протокол совместной работы

## Цель

Этот документ задает рабочий режим, в котором агент:

1. документирует, что было сделано;
2. фиксирует, когда это было сделано;
3. понимает, какой шаг завершен;
4. перед следующим шагом сначала сверяется со статусом;
5. предлагает следующий шаг на подтверждение пользователю;
6. только после подтверждения выполняет работу.

## Обязательные файлы

Перед началом каждого нового шага агент обязан прочитать:

1. `docs/roadmap/research-pipeline-backend-plan.md`
2. `docs/status/execution-status.md`
3. `docs/status/work-log.md`

После завершения каждого шага агент обязан обновить:

1. `docs/status/execution-status.md`
2. `docs/status/work-log.md`

## Рабочий цикл

Каждый шаг выполняется по одному и тому же сценарию:

1. Агент сверяет roadmap и текущий статус.
2. Агент пишет пользователю:
   - что уже сделано;
   - какой следующий шаг рекомендуется;
   - что именно будет сделано в этом шаге.
3. Пользователь:
   - подтверждает шаг;
   - или корректирует его.
4. Агент выполняет работу.
5. Агент документирует результат.
6. Агент обновляет статус проекта и предлагает следующий шаг.

## Триггер-фразы пользователя

Если пользователь пишет одну из фраз ниже, агент должен сначала проверить статус, а не сразу начинать произвольную новую работу:

1. `перейдем к следующему шагу`
2. `что дальше`
3. `какой следующий шаг`
4. `продолжай по плану`
5. `двигаемся дальше`

## Как агент должен отвечать на переход к следующему шагу

Формат ответа должен быть таким:

1. Кратко перечислить, какие шаги уже завершены.
2. Назвать следующий рекомендуемый шаг по `execution-status`.
3. Кратко описать результат, который будет получен.
4. Попросить подтверждение или корректировку.

Пример:

`Сверил статус и roadmap. Завершены шаги A0 и A1. Следующий рекомендуемый шаг: A2 — спецификация лейблов и evaluation plan. В этом шаге я подготовлю формат разметки valence/arousal, activity labels и правила валидации. Подтверди этот шаг или скорректируй его.`

## Как агент должен документировать завершение шага

После выполнения шага агент обязан:

1. отметить шаг как `completed` или другой актуальный статус в `execution-status`;
2. указать дату обновления;
3. коротко описать результат;
4. добавить запись в `work-log`.

## Формат записи в журнал

Каждая запись в `work-log` должна содержать:

1. дату;
2. идентификатор шага;
3. что запросил пользователь;
4. что было сделано;
5. какие файлы были изменены или созданы;
6. результат;
7. рекомендуемый следующий шаг.

## Разрешенные статусы шага

Используются только эти статусы:

1. `completed`
2. `in_progress`
3. `next`
4. `pending`
5. `blocked`
6. `cancelled`

## Правило минимального инкремента

Агент не должен перескакивать через крупные фазы. Следующий шаг должен быть:

1. небольшим;
2. проверяемым;
3. с понятным результатом;
4. привязанным к roadmap.

## Практическое правило для пользователя

Чтобы продолжать работу в этом режиме, достаточно писать:

1. `перейдем к следующему шагу`
2. `покажи что сделано и что дальше`
3. `выполняй следующий шаг`

Если пользователь хочет изменить порядок, он пишет:

1. `пропускаем этот шаг`
2. `сначала сделай другой шаг`
3. `скорректируй план`
