# Архитектурная схема monorepo

## Цель

Собрать backend в одном репозитории так, чтобы:

- сервисы развивались независимо;
- контракты были централизованы;
- инфраструктура и CI были единообразны;
- мобильный Swift-клиент оставался отдельным продуктовым репозиторием.

## Верхнеуровневая модель

```text
backend monorepo
  ├─ services      -> отдельные сервисы по бизнес-контекстам
  ├─ contracts     -> источник истины для API и событий
  ├─ platform      -> платформенный shared code
  ├─ infra         -> локальная и production-инфраструктура
  └─ docs          -> архитектура и решения
```

## Что считать отдельным сервисом

Сервис создается, если у модуля есть хотя бы часть из следующего:

1. Собственная бизнес-ответственность.
2. Отдельные данные и жизненный цикл.
3. Независимое масштабирование.
4. Собственные внешние интеграции.

Если этого нет, лучше начать с модуля внутри существующего сервиса, а не с нового микросервиса.

## Как связать с отдельным Swift-репозиторием

Рекомендуемый подход:

1. Backend хранит OpenAPI/event-контракты в `contracts/`.
2. CI генерирует Swift SDK или сетевые модели.
3. Swift-репозиторий подключает версионированный клиентский артефакт.

Так frontend не зависит от внутренней структуры backend monorepo.
